# План ускорения перевода: параллелизм + MLX + учёт свободной памяти

Цель: сократить время перевода (сейчас ~21 мин на 300 страниц) за счёт
параллельных запросов к модели и более быстрого рантайма MLX на Apple Silicon.
Ожидаемый эффект: этап 1 — 2–4×, этап 3 — ещё +20–40%.

Контекст кода:
- `engine/translator.py` — блокирующий последовательный `translate()` (один HTTP-запрос на блок).
- `engine/translate_pdf.py:107-118` — последовательный цикл перевода с прогрессом и ETA.
- `MacPDFTranslator/Sources/MacPDFTranslator/` — SwiftUI-клиент (OllamaClient, TranslationViewModel, ContentView, EngineRunner).

## Этап 1. Параллельный перевод через Ollama (наибольший выигрыш)

### 1.1 `engine/translator.py` — потокобезопасность и батч-метод
- [x] Сделать накопление статистики (`total_eval_count`, `total_eval_duration_ns`)
      потокобезопасным (`threading.Lock`).
- [x] Добавить метод `translate_batch(texts, src, dst, concurrency, on_progress)`:
      `concurrent.futures.ThreadPoolExecutor(max_workers=concurrency)`,
      результаты возвращаются **в исходном порядке** (по индексу, не as_completed).
      `on_progress(done_index, text_len)` вызывается из основного потока
      через сбор завершённых future (для JSON-событий прогресса).
- [x] Один `requests.Session` на поток (`threading.local`) — keep-alive соединения.
- [x] Ошибка в любом потоке → отмена остальных задач, проброс `OllamaError` наверх.
      Реализовано: 5 новых pytest-тестов с mock ThreadingHTTPServer
      (порядок, статистика, прогресс, ошибка, пустой список) — все проходят.
- Верификация: новый pytest с mock-сервером (потоковый `http.server`),
  проверяющий порядок результатов и агрегацию статистики; старые тесты проходят.

### 1.2 `engine/translate_pdf.py` — использование батч-метода
- [x] Заменить цикл строк 107–118 на `translate_batch` с эмиссией тех же
      событий `progress` (формат протокола не меняется — Swift-клиент не трогаем).
- [x] Новый аргумент CLI `--parallel N` (по умолчанию `0` = авто, см. этап 2;
      пока 0 → 4).
- [x] ETA при параллелизме: считать по фактической скорости символов/сек
      завершённых блоков (текущая формула остаётся корректной);
      `tps` из суммарной статистики — пометить в комментарии, что при
      параллелизме это агрегат, а не скорость одного потока.
- Верификация: e2e-прогон на `fixtures/` — результат идентичен по содержимому,
  время меньше; JSON-протокол валиден (существующий тест протокола).

### 1.3 Конфигурация сервера Ollama
- [ ] Ollama ≥ 0.2 сам батчит параллельные запросы (`OLLAMA_NUM_PARALLEL`,
      по умолчанию auto: 4 или 1 в зависимости от памяти). Проверять фактический
      параллелизм нельзя через API — поэтому: в README/плане задокументировать
      `launchctl setenv OLLAMA_NUM_PARALLEL 4` + перезапуск Ollama.
- [ ] Если сервер обслуживает запросы последовательно, код всё равно корректен
      (просто без ускорения) — деградация мягкая.

## Этап 2. Определение свободной памяти перед переводом

Зачем: выбрать безопасный уровень параллелизма (каждый параллельный слот
резервирует свой KV-кэш контекста) и не уронить Mac в свопе.

### 2.1 Новый модуль `engine/sysinfo.py`
- [x] `available_memory_gb()` — без новых зависимостей: парсинг `vm_stat`
      (free + inactive + purgeable страницы × page size). Запасной вариант —
      `psutil`, если парсинг окажется хрупким (тогда добавить в requirements).
- [x] `recommended_parallelism(available_gb, model_size_gb)` — эвристика:
      - модель уже загружена в память Ollama (узнать размер: `GET /api/ps`,
        поле `size_vram`/`size`);
      - резерв на слот ≈ 0.5–1 ГБ (KV-кэш типового контекста 4–8K);
      - итог: `clamp(1, 8, int((available_gb - 2) / резерв_на_слот))`,
        минимум 1, потолок 8.
- [x] В `translate_pdf.py` при `--parallel 0` (авто) вызывать эту логику и
      эмитировать информационное событие:
      `{"event":"info","parallel":N,"free_mem_gb":X}` (Swift-парсер игнорирует
      незнакомые события — проверено: EngineRunner пропускает строки,
      которые парсер вернул как nil, есть тест на unknown-событие).
- Верификация: unit-тест эвристики на табличных значениях; ручная проверка
  `vm_stat`-парсера на текущей машине.
  Выполнено: 10 новых тестов (vm_stat-парсер, эвристика, /api/ps + /api/tags
  через mock), весь набор движка — 25 passed. E2e на test.pdf: при 12.8 ГБ
  свободно и загруженной модели выбрано parallel=8, перевод 47 блоков
  на 4 страницах за ~34 с. Размер незагруженной модели берётся из /api/tags
  и вычитается из бюджета памяти.

## Этап 3. Бэкенд MLX (ЗАКРЫТ ПО РЕЗУЛЬТАТАМ БЕНЧМАРКА — 10.06.2026)

1. `gemma4:12b-mlx` (6.8 ГБ, Ollama) — отклонена: слишком тяжёлая для
   текущей памяти, съедает бюджет параллельных слотов.
2. In-process MLX (`mlx-lm`, модель `mlx-community/HY-MT1.5-1.8B-8bit`) —
   измерено скриптом `engine/bench_mlx_vs_ollama.py` на ../test.pdf
   (47 блоков, 14 814 симв.):
   - MLX последовательно: 58.4 с (254 симв/с), загрузка модели 1.1 с
   - Ollama bob-hymt parallel=8: 32.5 с (456 симв/с)
   MLX медленнее в 1.8× — однопоточный in-process generate() проигрывает
   батчингу 8 параллельных запросов Ollama; качество переводов сопоставимо.

Решение: остаёмся на Ollama + parallel=auto (этапы 1–2). Скрипты
`engine/test_mlx.py` и `engine/bench_mlx_vs_ollama.py` оставлены для
повторных замеров, если в mlx-lm появится батч-генерация.

Важно: «та же модель» в MLX — это те же веса, но другой формат файлов
(не GGUF из Ollama, а MLX-квантование из репозиториев `mlx-community`
на HuggingFace). Модель придётся скачать отдельно.

### 3.1 Сервер
- [ ] Зависимость `mlx-lm` в `engine/requirements.txt` (только Apple Silicon).
- [ ] Запуск: `mlx_lm.server --model mlx-community/<модель> --port 8080` —
      даёт OpenAI-совместимый API (`/v1/chat/completions`, `/v1/models`).

### 3.2 Новый `engine/mlx_translator.py`
- [ ] Класс `MLXTranslator` с тем же интерфейсом, что `OllamaTranslator`
      (`check()`, `translate()`, `translate_batch()`, `tokens_per_second()`),
      поверх `/v1/chat/completions`. Статистика токенов — из поля `usage`.
- [ ] В `translate_pdf.py`: аргументы `--backend ollama|mlx` (по умолчанию
      `ollama`) и `--mlx-url` (по умолчанию `http://localhost:8080`);
      фабрика выбора транслятора. Тот же запрет внешних адресов: только localhost.
- Верификация: pytest с mock OpenAI-совместимого ответа; e2e на fixtures
  с реально запущенным `mlx_lm.server`; бенчмарк «Ollama vs MLX» на одном
  и том же PDF — зафиксировать числа в этом файле.

### 3.3 Swift-клиент (минимальные правки)
- [ ] `ContentView` + `TranslationViewModel`: Picker «Бэкенд: Ollama / MLX»;
      для MLX — запрос списка моделей через `/v1/models` (расширить
      `OllamaClient` или добавить `MLXServerClient`).
- [ ] `EngineRunner`: пробрасывать `--backend`/`--mlx-url`/`--parallel`.
- Верификация: `swift build && swift test`; ручной прогон приложения.

## Что может сломаться (проверить после каждого этапа)

- Порядок переведённых блоков при параллелизме → тест порядка в 1.1.
- Точность ETA и поле `tps` при параллельных запросах → ручная проверка UI.
- JSON-протокол: новое событие `info` может уронить Swift-парсер → тест
  EngineRunner на незнакомое событие.
- Reasoning-модели (`<think>`) в MLX-ответах — фильтр `_THINK_RE` должен
  применяться и в `MLXTranslator`.
- Память: при завышенном параллелизме Ollama может выгружать/перегружать
  модель — следить за `/api/ps` во время бенчмарка.
- Гонки в накоплении статистики → lock в 1.1, проверяется тестом.
