1. Обзор продукта (Product Overview)
1.1. Цель продукта
Создание десктопного приложения для macOS, которое позволяет пользователям переводить PDF-документы с одного языка на другой с помощью локально запущенных языковых моделей (через Ollama). Ключевая ценность продукта — абсолютное сохранение исходного форматирования: структура, таблицы, изображения, шрифты и расположение текста в результирующем файле должны идеально соответствовать оригиналу.
1.2. Целевая аудитория
• Пользователи, работающие с конфиденциальными данными (юристы, аналитики, инженеры), которым запрещено загружать документы в облачные сервисы (Google Translate, DeepL).
• Специалисты, которым важна структура документа (инструкции, чертежи с текстом, презентации в PDF, научные статьи).
2. Архитектура и ключевые технологии (High-Level Architecture)
Для понимания логики работы приложения ниже представлена схема обработки документа:
1. Интерфейс (UI): SwiftUI (родной фреймворк для macOS).
2. Движок перевода: Локальный сервер Ollama (модели находятся /Users/dim/.ollama/models/manifests/registry.ollama.ai/library/bob-hymt).
3. Обработка PDF и сохранение структуры: * Для текстовых PDF: Использование библиотек парсинга (например, PDFKit или интеграция с Python-библиотеками через встроенный интерпретатор, такими как PyMuPDF / pdfplumber). • Для сканированных PDF (изображений): Встроенный в macOS движок Apple Vision Framework (OCR) для распознавания текста и координат его размещения. для примера можно посмотреть как это происходит в проекте '/Users/dim/vscodeproject/pdf->word'
3. Функциональные требования (Functional Requirements)
3.1. Управление файлами и настройки перевода (Главный экран)
Приложение должно содержать лаконичный интерфейс со следующими элементами управления:
• Выбор исходного файла: Кнопка «Обзор» (File Picker) с фильтром только для .pdf файлов. Поддержка Drag-and-Drop.
• Выбор выходного файла: Поле ввода имени и пути для сохранения результата. По умолчанию предлагается [Имя_Оригинала]_translated.pdf в той же папке.
• Выбор языка оригинала: Выпадающий список (Dropdown) с возможностью автоопределения (Auto-detect).
• Выбор языка перевода: Выпадающий список доступных языков. (английский, русский, китайский, немецкий, французский, итальянский, испанский, японский)
3.2. Требования к сохранению структуры (Layout Preservation) (английский, русский, китайский, немецкий, французский, итальянский, испанский, японский)
Это критически важное требование (Core Value). Приложение должно выполнять «Layout-Aware» перевод:
Элемент структуры	Требование к обработке
Текстовые блоки	Переведенный текст заменяет исходный строго в тех же координатах (Bounding Box).
Изображения	Извлекаются из оригинала и вставляются в результирующий PDF без изменений, на те же позиции.
Таблицы	Сетка таблицы (границы) сохраняется. Текст внутри ячеек переводится и центрируется/выравнивается аналогично оригиналу.
Шрифты и размер	Приложение должно пытаться сопоставить размер шрифта (Font Size) перевода так, чтобы текст не «вылезал» за границы исходного блока (динамический подбор reverse-scaling).
3.3. Мониторинг процессов (Progress & ETA)
Процесс перевода разделен на два этапа, для каждого из которых должен отображаться индивидуальный статус:
1. Этап 1: Распознавание и парсинг (OCR / Извлечение структуры) • Активируется всегда. Если PDF текстовый — процесс занимает секунды. Если PDF — скан, включается OCR. • Индикатор: Прогресс-бар (0-100%) + текстовый статус: Распознавание текста... Осталось примерно: ММ:СС.
2. Этап 2: Перевод (Взаимодействие с Ollama) • Построчный/поабзацный перевод через API Ollama (localhost:11434). • Индикатор: Прогресс-бар (по количеству страниц или текстовых блоков) + текстовый статус: Перевод текста... Осталось примерно: ММ:СС.
Алгоритм расчета времени (ETA): Вычисляется динамически. Для OCR — на основе среднего времени обработки одной страницы. Для перевода — на основе средней скорости генерации токенов локальной моделью (Tokens Per Second) помноженной на объем оставшегося текста.

5. Нефункциональные требования (Non-Functional Requirements)
5.1. Производительность и ресурсы
• Локальность: Приложение не должно отправлять тексты на внешние сервера. Вся работа с Ollama идет через localhost.
• Память: Поскольку обработка PDF больших объемов и работа с графикой нагружают RAM, приложение должно оптимизировать очистку кэша страниц (особенно при OCR).
5.2. Совместимость
• ОС: macOS 14.0 (Sonoma) и новее.
• Архитектура: Оптимизация под Apple Silicon (M1/M2/M3/M4) для быстрой работы Apple Vision OCR. 
5.3. Обработка ошибок (Error Handling)
• Если Ollama не запущена на компьютере, приложение должно выдать понятную плашку: «Ошибка подключения к Ollama. Убедитесь, что сервис запущен на порту 11434».
• Если переведенный текст длиннее оригинала и не влезает в Bounding Box, приложение должно автоматически уменьшать размер шрифта (минимальный порог — 6pt), чтобы не нарушить верстку.
6. Критерии приемки (Acceptance Criteria)
1. Тест 1 (Текстовый PDF): Документ на 10 страниц с 3 таблицами и 5 изображениями переведен полностью. Картинки на месте, сетка таблиц не съехала. Время перевода соответствует заявленному в ETA (погрешность не более 15%).
2. Тест 2 (Сканированный PDF): PDF, состоящий из картинок-отсканированных листов, успешно распознается через OCR, текст переводится и накладывается поверх оригинального изображения (или на чистую страницу с сохранением структуры).
3. Тест 3 (Автономия): При отключенном интернете приложение выполняет перевод в штатном режиме.
